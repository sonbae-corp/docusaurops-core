# How it works

DocusaurOps is an architecture pattern that any company can adopt to rationalize technical documentation across teams and projects — as a fully self-service experience. It is also a foundation for a customizable AI-SDLC approach, where teams can write their own skills using the same underlying techniques.

The solution is built on the following key components:

| Component | Role |
|-----------|------|
| **GitHub Copilot plugin** | Available in VS Code Chat and GitHub Copilot CLI |
| **Azure Application Gateway** | Routes incoming requests to the right documentation site |
| **Docusaurus** | Static site generator enabling a documentation-as-code approach |
| **GitHub Actions & reusable workflows** | Automates deployment end-to-end |
| **Copilot Connector** | Indexes all sites for AI-powered search (Enterprise Websites) |
| **WorkIQ** | Reads and writes Word documents; powers compliance document generation |

One of the goals of this solution is to demonstrate what it takes — from an architectural standpoint — for an AI-powered solution to be truly valuable, and how all components must work together to form a coherent whole.

### How it works

**1. Install the plugin**

Developers install the DocusaurOps plugin from their company marketplace — a dedicated GitHub repository configured to host the plugin definition. The plugin works in both VS Code Chat and GitHub Copilot CLI, so developers can choose the experience that suits them best.

> Short video here showing the installation process in both experiences.

**2. Scaffold a new documentation site**

From the plugin, developers trigger the **"Setup documentation"** command. They provide basic information — project name and desired URL slug — and the plugin takes care of the rest.

**3. Download and configure the template**

The `/docs-setup` skill clones the latest documentation structure from a dedicated template repository, maintained separately, using the GitHub CLI and the developer's own credentials. This separation of template from plugin means:
- The template can be updated independently, without touching the plugin.
- Changes are tracked with a full history and clear diffs.
- Migration paths are embedded directly in the template as instructions for the plugin (via a dedicated .md file).

An environment file is generated in the repository containing all deployment metadata (site name, slug, description, contact information, etc.), and all placeholder values are replaced automatically.

> Short video here showing the plugin in action and the resulting output.

**4. Write, preview, and deploy**

Developers write their documentation locally using the Docusaurus structure and preview it with the provided commands. When they push to the repository, the `deploy-docs.yml` workflow is triggered. It calls the reusable `deploy_site.yml` workflow from the shared repository, which handles the entire deployment:

- Configures Application Gateway backend pools and path rules to route traffic to the new site.
- Configures authentication on the new app.
- Deploys the site content from the caller repository.

**No credentials or variables need to be set up by the developer — everything is resolved automatically**.

> Short video here showing the deployment running in GitHub and the result on Azure.

**5. Site goes live and gets indexed**

The new site is published under the root URL `https://docs.my-company.com/{site_slug}` and automatically indexed by the Microsoft "Enterprise Websites" Copilot Connector, using the `sitemap.xml` dynamically generated by the root web application.

> Short video here showing the live site on Azure.

**6. Searchable across the entire company**

Using the `/docs-ask` skill and WorkIQ, developers can now ask questions across all projects — *"How was feature XYZ implemented in project ABC?"* — creating a mesh of shared knowledge across the entire organization.

:::tip A virtuous cycle
The more teams adopt the platform, the richer the shared knowledge becomes. Content added by one team becomes a reference for others, continuously improving quality across the organization — organically.
:::

## Compliance & traceability

    <img className="w-[60%]" src={ComplianceSkillsArchitecture}/>
</div>

Documentation is not just about explaining how a solution works. It is also a compliance and traceability artifact. DocusaurOps also addresses a common organizational pain point: **keeping formal documents made by non-technical stakeholders (like business analysts, product owners, etc.) in sync with the actual implementation**.

The goal is to foster collaboration between project stakeholders on documentation and specifications — specifically around two scenarios that appear in most organizations: **aligning requirements between business analysts and technical teams**, and **generating Software Design Documents for formal review**.

### `/docs-specs` — From requirements to workable issues

Business analysts or product owners often produce requirement documents that are not immediately actionable for technical teams as they often lack details. This skill bridges the gap by transforming raw requirements into structured, technically workable GitHub issues.

**1. Trigger the command**

A developer triggers the **"Generate specifications"** command from the plugin, pointing to a business requirement document in SharePoint or OneDrive, for instance a Word document (i.e. .docx).

**2. Extract and align requirements**

The plugin uses the WorkIQ Word MCP server to read the document and format its content according to a specification template. If information is missing, the plugin adds comments directly in the Word document so the analyst can respond. This back-and-forth continues until the requirements are complete enough to generate workable specifications.

**3. Create GitHub issues**

The finalized specifications are created as issues in the GitHub project associated with the repository — ready for the team to plan and implement.

**4. Automated PR review**

Once a PR is submitted, a GitHub cloud agent reviews the implementation against the associated requirement specification and the existing documentation. The agent adds feedback and comments directly in the PR to help developers align their implementation with the requirements.

---

### `/generate-sdd` — Produce a formal Software Design Document

An SDD is often required by external teams (enterprise architecture, security, compliance) to validate that a solution meets company standards. This skill generates a formal, well-structured SDD directly from the documentation-as-code content — no manual filling required.

**1. Trigger the command**

A developer triggers the **"Generate SDD"** command from the plugin.

**2. Extract content from the repository**

The plugin scans the existing documentation to extract all information needed to populate the SDD template.

**3. Generate the document**

A new Word document is created according to the template and stored in a dedicated SharePoint library for easy access and reference.

**Bonus — two-way synchronization:** The process also works in reverse. If a reviewer adds comments to the generated SDD (e.g., requesting more detail about the hosting infrastructure), the plugin reads those comments, creates the corresponding sections in the documentation site, and links them back in the SDD.

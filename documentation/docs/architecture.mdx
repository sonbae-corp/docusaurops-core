# Architecture

DocusaurOps is a resuable platform architecture any company can set up to rationalized technical documentation across all teams and projects. 

This is an foundation for an customizable approache (ex: writing your own skills using the same techniques). 
It is composed of the following components:

- A **GitHub Copilot plugin** to interact with users and trigger documentation site creation and search commands directly from their work environment (CLI or Visual Studio Code).
- Azure
Docusaurus
GitHub
WorkIQ 

The plugin itself is divided into mutliple skills set:

- Documentation setup
- Project specific document generation/revie

The flow is as follow:

1 - In a new or existing repostiory, developers install the DocusaurOps plugin from a company marketplace. In that case, the marketplace can be simply a dedicated GitHub repository configured as such hosting the plugin definition. The plugin can be instelled in either VS Code Chat experecne or GitHub Copilit CLI depending on the user prefernces.

> Short video here showing the installation process in both experiences.

2- From the plugin, developers trigger the "setup documentation" command to scaffold a new documentation structure for their project. The plugin asks for basic inforation such as the project name and the desired URL (i.e. slug). 

3. From here the /setup-docs skill kicks in:
    - It clones the latest documentation structure from a dedicated GitHub repository using the GitHub CLI (gh) using user credentials. Having a dedicated GitHub repository hosting the templates stucture allows to update it independently from the plugin, and push updates to it without the need to update the plugin itself. It also allows to keep track of changes and history on the template, and have a clear diff when changes are made. This repository contains specific instructions for the plugin (for instance the migration path is applicable) the Docausurus sturcutre and a predefined GitHub workflow template to deploy the documentation site.

> Short video here showing plugin work and outomes.

From the consumer repository, an environment file is created by the plugin containing all information needed to deploy the site (site name, slug, description, contact information, etc.) and files are updated accordingly. 

4. Developers write their documentation locally using the Docusaurus structure and preview it locally with the provided commands. Once done, they push to the repository, the `deploy-docs.yml` workflow is triggered. This workflow calls the reusable `deploy_site.yml` workflow hosted in a shared repository, passing all necessary variables to deploy the site on Azure. This workflow handles all the deployement process:
    - COnfigure the application gateway backend pools and path rules to route incoming requests to the new site.
    - COnfigure authetnication on the new app
    - Deploy the site content fron the caller repository

    In that process, developers do not need to setup any credentials or variables. All is done automatically for them!

> Short video here showing the deploymet done in GitHub and the outcomes on Azure.

5. The new site is published under the root URL `https://docs.my-company.com/{site_slug}` and automatically indexed by a "Enterprise website" Copilot connector (default connector from Microsoft) using the sitemap dynamically generated by the root web application.

> Short video here showing the site available on Azure.

6. Leveraging Copilot and WorkIQ, developers can then ask questions about other projects (how thinkgs were done in this XYZ project), creating a mesh knowledge network across the entire company!


The big benefit of this architecture it is a virtous cycle (it feeds itself). The more teams are using it, the more content is added to the general knowledge and can be used a reference for other projects improving the quality organicaly.

## How this works in practice

### 1. Install the plugin

Developers install the plugin on their Copilot CLI or directly in Visual Studio Code chat either manually by downloading the latest release from the GitHub repository, or directly from the GitHub marketplace once the plugin is available there.

### 2. Set up a documentation site

From CLI or chat, developers use the **"Setup documentation"** command from the plugin agent to create a new documentation site for their project. This triggers the following actions:

| Action | Detail |
|--------|--------|
| **Download template** | The latest content from the template repository is copied locally to the developer's machine using their current GitHub credentials. |
| **Create structure** | The documentation structure is created locally based on the template, or merged with existing documentation if any. The template contains an `INSTRUCTIONS.md` file with all required information about the structure and guidance to migrate existing content across versions. |
| **Create CI workflow** | A `deploy-docs.yml` workflow is created for CI, calling the reusable `deploy_site.yml` workflow in the `docusaurops-core` repository. |
| **Fill placeholders** | Placeholder values are replaced with user context (metadata, project title, description, etc.) and the user is prompted to fill in any missing information. |

### 3. Deploy on push

Whenever the content of the `/documentation` folder is updated, the deployment is triggered. The variables needed to deploy via `deploy_site.yml` are resolved directly from the `docusaurops-core` repository using a `DOCUSAUROPSREPO_TOKEN` secret set at organization level  no per-repository configuration needed. The documentation content is built in the context of the caller repository.

Each new site is provisioned via an **Azure deployment stack** (enabling rollbacks and idempotent redeployments). The following integration actions are performed automatically:

| Action | Detail |
|--------|--------|
| **Provision site** | Site is created via a deployment stack using a normalized template to ensure consistency (authentication, settings, etc.). |
| **Update gateway** | A new backend pool is added pointing to the new site, and a new path rule is added to route incoming requests to it. |
| **Update Entra ID app** | A redirect URL is dynamically added to the shared Entra ID app to ensure the OAuth2 authentication flow works correctly through the gateway. |
| **Update sitemap** | The site URL is added to `ENV_SITE_URLS`, used by the root web application to generate the `sitemap.xml` file consumed by the Copilot Connector. |

### 4. Site goes live

After provisioning completes, the new site is available at `https://docusaurops.ca/{site_slug}`.

### 5. Automatic indexing

The site is automatically indexed by the Copilot connector (Microsoft built-in "Enterprise Websites (On-Premises)") using the `sitemap.xml` generated by the root web application. This enables unified search across all sites using a single connector. Open Graph properties are indexed to provide rich answers with content previews in the agent.

### 6. Available in Copilot

All indexed sites are available through search or the Copilot Retrieval API to answer user questions when using the **"Ask documentation"** command from the plugin agent.

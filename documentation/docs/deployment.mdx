1. Create SPN in Entra ID.

In order for the solution to operate, you'll need to create two distinct SPN in your Azure Entra ID directory:

    - One used for the DocusaurOps site authentication (ex: **DocusaurOps - Auth**) . This SPN will be shared by all documentaion sites to authenticate users via EasyAuth.
      In the manifest, make sure the setting "Enable ID token" is enabled:

        ```json
        ...
        "web": {
            ...
            "implicitGrantSettings": {
                "enableAccessTokenIssuance": false,
                "enableIdTokenIssuance": true
            },
            ...
        },
        ...
        ```

    - One used for automated deployment from consumer repositories (ex: **DocusaurOps - Deploy**) This SPN requires to have the "Owner" role on the subscription as it will deploy resources via a deployment stack at subscription level. 
      Also, this SPN needs to be configured as the owner of the Authentication SPN (the previous one you created). This is required to be able to configure redirect URLs automatically for new deployed sites (i.e once a new site is added, a new redirect URL is added to the shared SPN to get the authentication flow work).

        Using Azure CLI, login as admin and run this command: `az ad app owner add --id <SPN_APP_ID_USED_FOR_AUTH> --owner-object-id <DEPLOYMENT_ENTEPRISE_APPLICATION_SPN_OBJECT_ID>`.

        The `--owner-object-id` here is the Object ID of the service principal used for deployment in **Enterprise Applications** (not the one in App Registrations). You can find it by going to Azure Portal > Entra ID > Enterprise Applications > Search for the SPN name > Overview.

        :::info Why not using a managed identity with federated crednetials for Azure infrastucture deployment from GitHub?
            Because federated cerencetials in managed identity must be created per repository and theand we don't know in advance which repository will use this feature, we can't crete them in advance.
            Having a client ID/secret ensure any exsiting ro new repo can actually deploy a site if needed.
        :::

        Then you need to add the API permission `Application.ReadWrite.OwnedBy` to this SPN and grant admin consent. This permission is required to allow the deployment SPN to update the authentication SPN with new redirect URLs when new documentation sites are deployed.

2. Create GitHub organization secrets.

In order fo consumers repositoriues to execute reusable workflow from the core repository, you need to create the following secrets in your GitHub organization. Organization secrets are shared by all repositories prevenintg consumers to exlicitly add them in their repository:

| Secret name | Description |
|-------------|-------------|
| `DOCUSAUROPS_REPO_TOKEN` | GitHub token with access to the `docusaurops-core` repository. This is required as the consumer repos will execute workdlows in the context of the core repository.     |
| `DOCUSAUROPS_AZURE_DEPLOY_CREDS` | Azure credentials to connect to the DEV tenant. Used in the `azure/login@v3` action. |
|

**To generate a repository token**:

In GitHub, go to your profile, settings->developer settings->personal access tokens->tokens (classic) and generate a new token with the following permissions:

- Read access to actions variables, code, environments, and metadata
- Read and Write access to actions

**To generate Azure credentials for GitHub Actions (for the `azure/login@v3` action)**:

```json
{
    "clientSecret":  "<client-secret-of-the-deployment-SPN>",
    "subscriptionId":  "<subscription-id>",
    "tenantId":  "<tenant-id>",
    "clientId":  "<client-id-of-the-deployment-SPN>"
}
```

Add the two secrets at the organization level in GitHub. This way, they will be automatically available in all repositories within the organization, including the consumer repositories that will use the reusable workflows from the core repository to deploy documentation sites:

3. Deploy the DocusaurOps core infrastructure in Azure

- Generate a self signed certificate.

This certificate is used by the application gateway to secure access to all documentations sites through HTTPS. For the purpose of this solution, we use a self signed certificate, but in a production environment, it is recommended to use a certificate from a trusted authority to ensure the security of your documentation sites:

:::important
The domain name should be in this format: `"docusaurops-<ENV_AZURE_ENV_NAME>.<ENV_AZURE_DEPLOY_LOCATION>.cloudapp.azure.com"` as it must match the hostname that the Application Gateway will expose.

Ex: `"docusaurops-my-env.canadacentral.cloudapp.azure.com"`
:::

```powershell
.\scripts\New-SelfSignedCertificate.ps1 -Domain "docusaurops-demotest.canadacentral.cloudapp.azure.com" -PfxPassword (ConvertTo-SecureString "bAfD2m9AdF*dh_*c6xx_" -AsPlainText -Force) -InstallRootCA
```

This command create install also the Root certificate in your machine. This way the site will be trusted when using your brwoser (otherwise, you'll need to add it manuaylll, especially on Google Chrime or Edge).
The scripts output the PFX certicate as base64 value. You need to use that value for the `ENV_AZURE_DOCUSAUROPS_CERT_BASE64_PFX` secrets in the next step. **After executing the script, the base64 value is automatically copied to your clipboard **.

:::important
The certificate pfx file is referenced by the `templates/main.bicep` when priviosiong the core infrastructure.
:::

- Configure variables and secrets on the core repository. This variable will be used to deploy the core infrastructure in Azure

Variables:

| Variable name | Description | Example |
|---|---|---|
| `ENV_AZURE_ENV_NAME` | Short environment name used in resource naming. | `my-env` |
| `ENV_AZURE_DEPLOY_LOCATION` | Azure region where the platform will be deployed. | `canadacentral` |
| `ENV_AZURE_DEPLOY_TENANT_ID` | Microsoft Entra tenant ID used for deployment authentication. | `00000000-0000-0000-0000-000000000000` |
| `ENV_AZURE_DEPLOY_APP_CLIENT_ID` | Client ID of the service principal used for automated deployments. | `11111111-1111-1111-1111-111111111111` |
| `ENV_AZURE_DEPLOY_APP_CLIENT_SECRET` | Client secret for the deployment service principal. Store this securely and never commit a real value. | `<deployment-spn-secret>` |
| `ENV_AZURE_DEPLOY_SUBSCRIPTION_ID` | Azure subscription ID that hosts the platform resources. | `22222222-2222-2222-2222-222222222222` |
| `ENV_AZURE_DEPLOY_STACK_ENV_NAME` | Logical stack name, usually derived from the environment stage. | `docusaurops-dev` |
| `ENV_AZURE_DEPLOY_RESOURCE_PREFIX` | Prefix applied to generated Azure resource names. | `docusaurops` |
| `ENV_AZURE_DEPLOY_STACK_RG_NAME` | Resource group name for the deployment stack. | `rg-docusaurops-dev` |
| `ENV_AZURE_DOCUSAUROPS_ENTRA_ID_CLIENT_ID` | Client ID of the Entra application used by the DocusaurOps site. | `44444444-4444-4444-4444-444444444444` |
| `ENV_AZURE_DOCUSAUROPS_ENTRA_ID_CLIENT_SECRET` | Client secret for the DocusaurOps Entra application. Store this securely and never commit a real value. | `<docusaurops-app-secret>` |
| `ENV_AZURE_DEPLOY_STACK_RBAC_PRINCIPALS` | JSON array of objects defining additional RBAC principals (users or service principals) to be assigned roles on the deployed resources. Each object should have `principalId` and `principalType` properties. | `[{"principalId":"33333333-3333-3333-3333-333333333333","principalType":"ServicePrincipal"},{"principalId":"55555555-5555-5555-5555-555555555555","principalType":"User"}]`. For SPN, you must use the Object ID of the Enterprise application (not the app registration). For user type, this is simply the user ID in Entra |

Secrets:

| Secret name | Description | Example |
|---|---|---|
| `ENV_AZURE_DOCUSAUROPS_CERT_BASE64_PFX` | Base64-encoded PFX certificate data for the Application Gateway SSL certificate. This should be the output from the certificate generation step. Store this securely as GitHub secert and never commit a real value. | `<base64-encoded-pfx-certificate>` |
| `ENV_AZURE_DOCUSAUROPS_CERT_PASSWORD` | Password protecting the exported PFX certificate used by the gateway. Store this securely and never commit a real value. | `<certificate-password>` |

In the GitHub repository, go to Actions and run the `Deploy DocusaurOps Core Infrastructure` workflow of the main branch.

When the deployment is done, you need to add a new variable `ENV_AZURE_DOCUSAUROPS_ROOT_WEBAPP_NAME` in the core repository. This variable corresponds to the web application name of the root site and used by the deploy site script to update the app settings for the sitemap:

| Variable name | Description | Example |
|---|---|---|
| `ENV_AZURE_DOCUSAUROPS_ROOT_WEBAPP_NAME` | The name of the root web application deployed in Azure. This is used to update the sitemap configuration with the correct URL of the root site. | `app-docusaurops-root-qlrnl` |
